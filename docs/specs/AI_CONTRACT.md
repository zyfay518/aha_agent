# AI 图片识别与穿搭生成契约

> 版本：0.1  
> 状态：阶段 0 验收候选

## 1. 原则

- 模型输出不直接写入数据库。
- 所有模型输出必须通过 JSON Schema 校验。
- 用户确认字段优先于 AI 推断字段。
- 模型不能决定数据权限、所有权或是否允许购买建议。
- 原图只在完成当前任务所需范围内发送给模型。
- 无法判断时返回不确定，不伪造品牌、材质或具体款式。

## 2. 图片识别任务

目标：判断图片是否适合录入，并生成最少候选字段。

输入：

- 一张用户上传图片。
- 允许的大类和二级分类列表。
- 标准颜色词表。

输出 Schema：

```json
{
  "is_wearable_item": true,
  "item_count": 1,
  "candidate": {
    "name": "白色长袖衬衫",
    "category": "top",
    "subcategory": "shirt",
    "primary_color": "white",
    "secondary_color": null,
    "season_tags": ["spring", "autumn"],
    "style_tags": ["basic", "casual"]
  },
  "confidence": {
    "category": 0.96,
    "subcategory": 0.83,
    "primary_color": 0.94
  },
  "warnings": []
}
```

校验规则：

- `item_count` 只允许 0、1、2；2 表示“多件或无法可靠分离”。
- 非穿搭单品时 `candidate` 为 null。
- `category` 必须属于四大类。
- `subcategory` 和颜色必须属于当前白名单，无法判断使用 `other`/`unknown`。
- 置信度为 0–1，仅用于 UI 和监控，不作为权限依据。
- 名称不得包含模型猜测的品牌。

## 3. 穿搭需求解析

MVP 只抽取轻量条件：

```json
{
  "style_terms": ["simple", "comfortable"],
  "occasion": "daily",
  "color_terms": [],
  "required_categories": ["top", "bottom", "shoes"],
  "notes": []
}
```

未提及的信息保持空值，不追问身材、肤色和复杂偏好。请求模糊时默认日常、简单、舒适。

## 4. 穿搭生成职责划分

确定性代码负责：

- 查询当前用户单品。
- 类别完整性。
- 颜色基础兼容。
- 组合去重。
- 校验单品所有权和删除状态。
- 判断是否存在可接受的已有单品组合。
- 决定是否允许输出衣橱缺口。

模型负责：

- 对候选组合做轻量语义排序。
- 生成不超过约 60 个汉字的说明。
- 在服务端允许缺口时描述通用单品规格。

模型不得返回数据库中未提供的 item ID。

## 5. 缺口输出

```json
{
  "category": "shoes",
  "reason": "当前衣橱没有鞋履，无法组成完整搭配。",
  "suggested_spec": {
    "subcategory": "sneakers",
    "colors": ["white", "beige"],
    "notes": "选择低饱和、易与现有上装和下装组合的款式。"
  },
  "urgency": "optional"
}
```

禁止内容：

- 具体商品和购买链接。
- 虚构用户拥有的衣物。
- “必须买”“立即购买”等推动冲动消费的语气。
- 在已有组合合格时仍输出缺口。

## 6. 失败与降级

- 图片分析失败：允许用户重试或手动填写，不丢失上传。
- 结构化输出校验失败：最多进行一次受控修复请求，仍失败则返回可恢复错误。
- 穿搭语义排序失败：使用确定性评分结果。
- 说明生成失败：使用模板说明。
- 模型超时或限流：不重复创建记录，向用户显示稍后重试。

## 7. 质量评估集

建立固定、可授权使用的测试集：

- 每个大类至少 10 张单件图片。
- 白、黑、灰、蓝、棕等常见颜色覆盖。
- 平铺图和衣架图覆盖。
- 多件图片、非衣物图片和模糊图片作为负例。
- 多种衣橱组成和缺类情况的结构化样例。

记录指标：大类准确率、主色准确率、多件检测召回率、用户字段修改率、已有单品推荐率和错误缺口触发率。
