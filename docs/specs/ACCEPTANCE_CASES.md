# 阶段 0 验收案例基线

> 版本：0.1  
> 用途：阶段 1–9 的开发与回归测试依据

## 1. 账号与隔离

| 编号 | 场景 | 预期结果 |
|---|---|---|
| AUTH-01 | 未登录访问衣橱 | 跳转登录或返回 401 |
| AUTH-02 | A 查询自己的单品 | 成功 |
| AUTH-03 | A 使用 B 的 item ID 查询 | 返回未找到，不泄露存在性 |
| AUTH-04 | A 修改请求中伪造 B 的 user_id | 被忽略或拒绝 |
| AUTH-05 | 用户撤销 ChatGPT 授权 | 后续 MCP 调用失败，网页数据保留 |

## 2. 图片录入

| 编号 | 场景 | 预期结果 |
|---|---|---|
| ITEM-01 | 上传清晰白色衬衫平铺图 | 进入确认，候选为上装/衬衫/白色 |
| ITEM-02 | 上传衣架上的长裤 | 进入确认，候选为下装 |
| ITEM-03 | 上传两件衣服合照 | 提示一次只上传一件，不正式保存 |
| ITEM-04 | 上传非衣物图片 | 拒绝录入并允许更换 |
| ITEM-05 | AI 识别错误后用户修改 | 保存用户确认值 |
| ITEM-06 | 对同一 upload_id 连续确认两次 | 只创建一个单品 |
| ITEM-07 | AI 服务超时 | 保留临时状态，可重试或手填 |
| ITEM-08 | AI 返回 429 | 重试一次后显示额度/繁忙提示，仍可通过标签保存 |
| ITEM-09 | 保存按钮连续点击 | 按钮立即禁用，且同一 upload_id 最多创建一个单品 |
| ITEM-10 | 结构化记录成功但图片绑定失败 | 不宣告保存成功；重试绑定或回滚不完整记录 |
| ITEM-11 | 上传商品页截图 | 展示图只保留服装主体，移除界面、价格、人物和场景 |
| ITEM-12 | 上传透明或非方形处理图 | 服务端统一输出 1200×1200 纯白背景 JPEG |

## 3. 衣橱管理

| 编号 | 场景 | 预期结果 |
|---|---|---|
| WARD-01 | 用户有四类共 28 件 | 摘要总数与分类之和一致 |
| WARD-02 | 按鞋履筛选 | 只返回未删除鞋履 |
| WARD-03 | 修改名称、季节和双颜色 | 刷新后保持新值 |
| WARD-04 | 删除一件单品 | 立即从列表和新推荐中消失 |
| WARD-05 | 访问私有图片 | 仅有效签名链接可读取 |
| WARD-06 | 用户连续两次请求“看衣橱” | 返回完全相同的用户专属链接，页面数据随衣橱更新 |
| WARD-07 | 用户只说“看衣橱” | 只返回可点击衣橱链接，不追加文字清单或解释 |
| WARD-08 | 检查公开衣橱 URL | URL 不含 `AHA-...` 操作访问码，只包含只读 UUID |
| WARD-09 | 把只读 UUID 当作 Agent 访问码调用写工具 | 返回 `INVALID_ACCESS_CODE`，数据不变 |
| WARD-10 | 使用不存在的只读 UUID | 页面返回 404，不泄露任何衣橱信息 |
| WARD-11 | 打开拥有多个类别单品的图片衣橱 | 页面按上装、下装、鞋履、包袋显示分类入口、分类数量和独立分区 |
| WARD-12 | 某一分类暂时为空 | 分类仍可见且数量为 0，分区显示简短空状态 |

## 4. 穿搭推荐

| 编号 | 场景 | 预期结果 |
|---|---|---|
| OUTFIT-01 | 上装、下装、鞋履充足 | 返回 1–3 套，全部来自现有衣橱 |
| OUTFIT-02 | 用户要求“简单舒服” | 不因信息少而强制追问 |
| OUTFIT-03 | 完全没有鞋履 | 可指出鞋履缺口，语气为可选补充 |
| OUTFIT-04 | 有合格组合 | `gap` 必须为 null |
| OUTFIT-05 | 模型返回陌生 item ID | 服务端拒绝该结果 |
| OUTFIT-06 | 某件单品刚被删除 | 新结果不得包含它 |
| OUTFIT-07 | 生成三套 | 结果之间存在实质差异，不只是顺序不同 |
| OUTFIT-08 | 模型服务失败 | 使用确定性结果或明确可重试，不丢失输入 |
| OUTFIT-09 | Agent 调用 `create_outfit_board` | 工具结果的 `content` 直接包含 `image/jpeg` 图片，正常回复不再要求用户打开网页链接 |
| OUTFIT-10 | 穿搭板包含两件已有单品 | 图片使用衣橱中对应白底主体图，名称正确，消息内容不包含访问码或穿搭链接 |

## 5. 减少冲动消费

| 编号 | 场景 | 预期结果 |
|---|---|---|
| BUY-01 | 用户想买与已有单品相似的黑衬衫 | 先展示已有相似单品和可搭组合 |
| BUY-02 | 用户衣橱类别齐全 | 不主动提供商品或购物链接 |
| BUY-03 | 确实缺少必要类别 | 只描述通用规格，不推荐具体商品 |
| BUY-04 | 衣橱缺口可被非购物方案解决 | 优先给出现有替代方案 |

## 6. 删除和生命周期

| 编号 | 场景 | 预期结果 |
|---|---|---|
| DELETE-01 | 删除单品 | 设置软删除时间并立即隐藏 |
| DELETE-02 | 清理任务重复运行 | 结果幂等，无异常副作用 |
| DELETE-03 | 用户请求删除账号 | 立即停止正常使用并撤销授权 |
| DELETE-04 | 恢复期结束 | 删除业务数据和图片 |
| DELETE-05 | 临时上传超过有效期 | 图片和临时记录被清理 |

## 7. ChatGPT App

| 编号 | 场景 | 预期结果 |
|---|---|---|
| APP-01 | 用户问衣橱摘要 | 调用摘要工具而非加载全部图片 |
| APP-02 | 用户说“删掉它”但对象不唯一 | 要求选择，不执行删除 |
| APP-03 | 用户确认具体删除对象 | 调用删除工具并返回结果 |
| APP-04 | 新对话中使用已授权 App | 可继续查询当前用户衣橱 |
| APP-05 | 授权过期或撤销 | 引导重新连接，不暴露内部错误 |
| APP-06 | 新增单品后图片绑定成功 | 同一轮保存结果可立即在图片衣橱中看到白底主体图 |

## 8. 阶段 0 通过条件

阶段 0 文档验收通过需要确认：

1. 页面和对话流程覆盖 MVP 全部功能。
2. 数据表足以支持录入、管理、推荐、邀请和长期授权。
3. RLS 与私有存储边界清楚。
4. Web API、MCP 工具和 AI 输出职责不重叠。
5. 已有单品优先规则可以被确定性测试。
6. 错误、重试、删除和跨用户访问均有预期行为。
7. 未把任何延期功能偷偷带入 MVP。
