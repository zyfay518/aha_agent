### 2026-07-12 — 建立 MVP 产品基线

- 变更类型：初始定义。
- 目标用户：衣服较多但不知道如何搭配，并希望减少冲动消费的人。
- 产品重心：衣橱管理和减少购物，而非专业造型。
- 使用方式：个人工具，优先支持 ChatGPT。
- 推荐结果：1–3 套单品卡片拼图，不生成真人上身图。
- 录入方式：一张照片只包含一个单品，优先支持平铺图和衣架图。
- 分类范围：上装、下装、鞋履、包袋；暂不支持复杂配饰。
- 用户偏好、自定义分类、衣物状态、分享和商业化均延期。
- 数据：图片持续保存，用户可以删除；账号删除必须进入可恢复的短期软删除及最终清除流程。
- 邀请：接受“一次确认连接”代替“纯文本码无确认自动安装”。
- 平台迁移：核心后端和 MCP 保持平台无关，ChatGPT App 作为适配层。

## 5. 架构决策记录

### ADR-001：核心服务与 ChatGPT 适配层分离

- 日期：2026-07-12
- 状态：已接受。
- 决策：用户、衣橱、图片、识别和推荐逻辑独立于 ChatGPT；ChatGPT 通过 MCP 调用核心服务。
- 原因：降低未来迁移到其他 Agent 平台的成本。
- 影响：业务规则不能只写在 ChatGPT 提示词中，必须在服务端校验。

### ADR-002：MVP 使用卡片拼图，不做虚拟试穿

- 日期：2026-07-12
- 状态：已接受。
- 决策：使用用户上传的单品图片组成静态或交互式拼图。
- 原因：更快验证衣橱管理和穿搭推荐的真实需求。
- 影响：暂不建设人物生成、身体建模或服装形变能力。

### ADR-003：购买建议是受限制的降级结果

- 日期：2026-07-12
- 状态：已接受。
- 决策：只有已有衣物无法形成合理搭配时，才允许返回衣橱缺口。
- 原因：保持减少冲动消费的核心价值。
- 影响：推荐接口必须明确返回是否仅使用已有单品，并记录缺口触发原因。

### ADR-004：邀请凭证与长期授权分离

- 日期：2026-07-12
- 状态：已接受。
- 决策：邀请码只用于进入连接流程；长期访问使用可撤销的 OAuth 授权。
- 原因：防止邀请码泄漏后直接暴露用户衣橱。
- 影响：需要邀请服务、OAuth、refresh token 和撤销能力。

## 6. 风险与待确认事项

| 风险或事项 | 当前处理方式 | 触发复审的时间 |
|---|---|---|
| ChatGPT App 发布和计划权限可能变化 | 先在支持的开发环境测试，核心能力保持 MCP 化 | 开始阶段 7 时 |
| 多件衣物检测可能不稳定 | MVP 以提示用户为主，不做自动拆分 | 阶段 2 验收时 |
| 色彩和风格评分主观 | 先建立基础规则并收集真实反馈 | 阶段 4 内测时 |
| 用户录入成本过高 | 控制确认字段，只让用户处理名称、分类和主色 | 阶段 2 验收时 |
| 图片包含私人环境信息 | 明确删除机制，后续评估自动裁剪和去背景 | 阶段 2、9 |
| 数据和模型费用 | MVP 记录调用量与图片存储量，不提前商业化 | 内测后 |
