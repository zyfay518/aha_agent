### 2026-07-12 — 手机截图自动压缩需求

- 测试图片 `IMG_3222.PNG` 为 1206×2622、约 4.34MB，略高于原定 4MB 上限。
- 产品判断：不应要求普通用户理解图片体积或手动压缩手机截图。
- 调整：浏览器端对较大图片和 PNG 自动缩放到最长边 1800px，并转换为质量 0.86 的 JPEG 后上传。
- 服务端继续保留 4MB、MIME 类型和身份校验，作为安全边界。
- 发现分类精度问题：原二级分类只能将泳裤归入普通短裤，无法保留实际用途。
- 调整：增加 `swimwear`（泳裤）二级分类；识别提示明确忽略商品截图里的缩略图、模特示意图、价格和界面元素，并使用高细节视觉输入。
- 压缩增加结果体积校验和逐级质量压缩，确保提交文件低于服务端安全上限。
- 预期识别：黑色泳裤/双层泳裤；大类 `bottom`；二级分类 `swimwear`；主色 `black`。

### 2026-07-12 — 首次真实上传验收问题修复

- 用户真实上传成功，但 OpenAI 返回 HTTP 429，上传记录错误码为 `OPENAI_429`；图片本身已正常写入私有存储。
- AI 请求对 429 增加一次短暂重试，并保存经过清理的错误类型，界面明确区分额度/繁忙与普通识别失败。
- 确认界面由下拉框改为三组点击标签：类别单选、季节多选、颜色最多双选；二级分类由 AI 或系统保留，不要求用户操作。
- 原保存流程依赖先移动 Storage 对象再插入记录，容易产生中间失败。调整为直接引用已上传的私有图片路径、利用 `source_upload_id` 唯一约束实现幂等保存，并禁止保存按钮重复提交。
- 查询上传记录时显式附带 `user_id` 条件，与 RLS 所有权规则保持一致。

### 2026-07-12 — 建立 MVP 产品基线

- 变更类型：初始定义。
- 目标用户：衣服较多但不知道如何搭配，并希望减少冲动消费的人。
- 产品重心：衣橱管理和减少购物，而非专业造型。
- 使用方式：个人工具，优先支持 ChatGPT。
- 推荐结果：1–3 套单品卡片拼图，不生成真人上身图。
- 录入方式：一张照片只包含一个单品，优先支持平铺图和衣架图。
- 分类范围：上装、下装、鞋履、包袋；暂不支持复杂配饰。
- 用户偏好、自定义分类、衣物状态、分享和商业化均延期。
- 数据：图片持续保存，用户可以删除；账号删除必须进入可恢复的短期软删除及最终清除流程。
- 邀请：接受“一次确认连接”代替“纯文本码无确认自动安装”。
- 平台迁移：核心后端和 MCP 保持平台无关，ChatGPT App 作为适配层。

## 5. 架构决策记录

### ADR-001：核心服务与 ChatGPT 适配层分离

- 日期：2026-07-12
- 状态：已接受。
- 决策：用户、衣橱、图片、识别和推荐逻辑独立于 ChatGPT；ChatGPT 通过 MCP 调用核心服务。
- 原因：降低未来迁移到其他 Agent 平台的成本。
- 影响：业务规则不能只写在 ChatGPT 提示词中，必须在服务端校验。

### ADR-002：MVP 使用卡片拼图，不做虚拟试穿

- 日期：2026-07-12
- 状态：已接受。
- 决策：使用用户上传的单品图片组成静态或交互式拼图。
- 原因：更快验证衣橱管理和穿搭推荐的真实需求。
- 影响：暂不建设人物生成、身体建模或服装形变能力。

### ADR-003：购买建议是受限制的降级结果

- 日期：2026-07-12
- 状态：已接受。
- 决策：只有已有衣物无法形成合理搭配时，才允许返回衣橱缺口。
- 原因：保持减少冲动消费的核心价值。
- 影响：推荐接口必须明确返回是否仅使用已有单品，并记录缺口触发原因。

### ADR-004：邀请凭证与长期授权分离

- 日期：2026-07-12
- 状态：已接受。
- 决策：邀请码只用于进入连接流程；长期访问使用可撤销的 OAuth 授权。
- 原因：防止邀请码泄漏后直接暴露用户衣橱。
- 影响：需要邀请服务、OAuth、refresh token 和撤销能力。

### 2026-07-14 — 公开衣橱链接与操作访问码分离

- 验收发现固定衣橱链接直接包含 `AHA-...` 操作访问码，Agent 因敏感凭证泄露风险拒绝显示。
- 新增每用户稳定、可撤销的只读 `view_id`；公开衣橱和穿搭链接只包含 UUID，不再包含 Agent 操作访问码。
- 只读 UUID 仅可列出对应衣橱和读取对应图片，不能用于新增、修改、删除或图片写入；伪造 UUID 返回 404。
- 旧的访问码式公开页面停止工作，避免历史模式继续暴露读写凭证。
